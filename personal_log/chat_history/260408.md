这个仓库本质上是一个“RoboDual 主项目 + CALVIN 仿真/数据子项目 + 本地模型权重与验证脚本”的组合。核心目标是做机器人操作里的双系统策略：
  慢速的通才模型负责给出泛化的粗动作和语义条件，快速的专家模型负责实时、细粒度控制。

  顶层结构

  - RoboDual/：主实现，训练、评测、模型库基本都在这里。
  - calvin/：CALVIN 基准环境和数据工具，是 RoboDual 训练/评测依赖的外部子项目。
  - models/：本地模型权重。
      - models/generalist/ 是 Hugging Face 风格的通才模型目录，里面有 config.json、modeling_prismatic.py、分片 safetensors、tokenizer
        等。
      - models/specialist/ 里是专家策略权重 Specialist+Depth+Gripper.pt。
  - evaluation_results/：评测输出结果。
  - 根目录 smoke_models.py、smoke_dualsystem.py：最小化加载测试脚本，用来验证通才模型、专家模型和双系统能否正常加载。
  - 其他如 codex_thread.md、conda_run_probe.txt 更像工作痕迹，不属于核心业务代码。

  RoboDual/ 里最重要的部分

  - RoboDual/README.md：项目说明，直接把仓库分成通才训练、专家训练、CALVIN 评测三条主线。
  - vla-scripts/：真正的任务入口。
      - RoboDual/vla-scripts/train_generalist_calvin.py：训练通才模型。它加载 OpenVLA/Prismatic 风格的视觉语言动作模型，把 CALVIN 轨迹离
        散化成动作 token，然后做监督微调。
      - RoboDual/vla-scripts/train_spacialist_calvin.py：训练专家模型。里面的 DualSystem 类是关键，它先跑慢系统拿到粗动作和隐藏状态，再
        把这些当条件喂给扩散策略，训练快速专家去预测更细的动作序列。
      - RoboDual/vla-scripts/evaluate_calvin.py：CALVIN 长序列评测入口，负责多任务序列 rollout、统计成功率、保存结果。
      - RoboDual/vla-scripts/dual_sys_evaluation.py：双系统在评测时的核心控制逻辑。这里实现了“慢系统每隔几步更新一次，高速系统每步都出动
        作”，并且做了 temporal ensemble 来平滑动作输出。
      - RoboDual/vla-scripts/calvin_env_wrapper.py：对 CALVIN 环境做一层包装，让评测代码直接拿到原始观测和动作接口。
      - extern/：权重转换、校验脚本。
      - enrich_lang_annotations.json：语言增强/替换用的标注集合。
  - prismatic/：模型和数据的底层库，几乎是整个项目的“框架层”。
      - models/vlms/：通用视觉语言模型骨架。
          - RoboDual/prismatic/models/vlms/prismatic.py：定义 PrismaticVLM，负责把视觉 backbone、LLM backbone 和 projector 组起来。
      - models/vlas/：
          - RoboDual/prismatic/models/vlas/openvla.py：在 PrismaticVLM 上再包一层，加入动作 token 的解码和动作反归一化，变成真正的 VLA
            模型。
      - models/policy/：专家策略实现。
          - RoboDual/prismatic/models/policy/diffusion_policy.py：专家模型主体 DiffusionDiTImagePolicy。它把图像、深度、夹爪视角、触觉、
            通才隐藏状态、参考动作等条件编码后，用扩散模型生成动作序列。
          - diffusion_transformer.py：扩散 Transformer 主干网络。
          - vit.py：深度图/触觉图用的 ViT 编码器。
          - transformer_utils.py：Transformer 辅助模块。
      - vla/datasets/：
          - RoboDual/prismatic/vla/datasets/calvin_dataset.py：CALVIN 数据集加载器。它把单帧/多帧 npz 轨迹拼起来，取 RGB、深度、夹爪图、
            触觉、机器人状态、语言指令，再同时组织出通才和专家各自需要的输入。
          - RoboDual/prismatic/vla/action_tokenizer.py：把连续动作离散成 token，供 OpenVLA 训练和解码使用。
      - util/：
          - RoboDual/prismatic/util/data_utils.py：batch 拼接、padding、collator 等通用数据工具。
      - conf/：模型/VLA 配置注册表。
      - training/：训练策略、指标、分布式训练封装。
      - preprocessing/：数据准备和下载。
      - extern/hf/：给 Hugging Face AutoModel / AutoProcessor 适配的实现。
      - overwatch/：日志封装。
  - experiments/robot/：偏实验和跨环境评测。
      - openvla_utils.py、robot_utils.py：加载模型、处理图像、获取动作的通用工具。
      - libero/：LIBERO 环境评测脚本，说明这套方法不只打 CALVIN，也尝试迁移到其他机器人基准。
  - assets/：论文插图、框架图、时间线图。
  - CALVIN_ABC-D/：CALVIN 数据划分相关文件。
  - scripts/：通用训练/预处理/生成脚本。

  calvin/ 这部分在做什么

  - 这是一个基本完整拷贝的 CALVIN 项目，不是 RoboDual 自己从零写的环境。
  - calvin_env/：仿真环境本体。
      - envs/：PlayTable 环境和任务定义。
      - camera/：静态相机、夹爪相机、触觉传感器。
      - robot/：机械臂、IK、控制逻辑。
      - scene/：桌面场景和物体。
      - io_utils/、datarenderer.py、vrdatacollector.py：数据采集和渲染。
  - calvin_models/calvin_agent/：CALVIN 官方基线 agent 框架。
      - datasets/：数据模块。
      - evaluation/：多步/单步评测逻辑。
      - models/：基线模型和 CalvinBaseModel 接口。
      - training.py：Hydra + PyTorch Lightning 训练入口。
  - dataset/：数据目录。
  - scripts/、slurm_scripts/：可视化和集群运行辅助脚本。

  如果按“组件职责”来理解，主流程是这样

  1. DiskCalvinDataset 从 CALVIN 轨迹里读出多模态观测和动作。
  2. ActionTokenizer 把连续动作离散成 token，供通才模型训练。
  3. 通才模型在 train_generalist_calvin.py 中学习“图像 + 指令 -> 粗动作 token 序列”。
  4. 专家模型在 train_spacialist_calvin.py 中读取通才输出的粗动作和隐藏状态，把它们当条件，再结合更细粒度视觉/深度/夹爪输入，用扩散策略
     预测真实动作块。
  5. 评测时 DualSystemCalvinEvaluation 会周期性调用慢系统更新高层条件，并在每一步让快系统输出控制动作，最后在 CALVIN 环境里执行并统计成
     功率。

  整体上，这个项目不是普通的 Web/业务项目，而是一个机器人学习研究代码库：RoboDual 负责算法和模型，calvin 负责仿真环境与基准，models/ 负
  责本地权重，根目录脚本负责快速验证。整个仓库的组织方式也是围绕“训练通才、训练专家、组合评测”这三件事展开的。


› 那么现在如果我要实现：在自己设计的仿真环境（建模设置任务要求）里面应用它的代码结构，我的最快切入点是怎样的，请在对话框里回答我


• 最快的切入点，不是先改模型，而是先把你的仿真环境“伪装成它已经认识的输入/输出格式”。

  如果你的目标是“先跑起来”，最短路径是这条：

  1. 先只做“评测接入”，不要一开始就改训练。
  2. 仿照 RoboDual/vla-scripts/calvin_env_wrapper.py 写一个你自己的环境包装器。
  3. 让你的环境每一步返回和 RoboDual/vla-scripts/dual_sys_evaluation.py 里 step() 期望的一样的观测字典。
  4. 先用现成权重跑通推理闭环，再考虑数据集和再训练。

  它当前最核心的观测接口基本是：

  obs = {
      "rgb_obs": {
          "rgb_static": ...,
          "rgb_gripper": ...,
      },
      "depth_obs": {
          "depth_static": ...,
          "depth_gripper": ...,
      },
      "robot_obs": ...,
  }

  再加上一条语言指令 instruction，以及 step(action) 能执行 7 维动作。

  你可以把它理解成：
  DualSystemCalvinEvaluation.step(obs, instruction, step_id) 已经是一个现成的“策略控制器”，你要做的只是把你自己的环境接到它前后两端。

  为什么这是最快路径

  - 模型侧大部分代码你都不用动。
  - 双系统调度逻辑已经在 RoboDual/vla-scripts/dual_sys_evaluation.py 里写好了。
  - 环境适配层很薄，主要就是 reset / step / get_obs / get_info。
  - 这样你能最快验证：你的任务建模、相机视角、动作定义，是否和现有策略兼容。

  如果你的环境和 CALVIN 差别比较大，我建议分两阶段
  第一阶段：只接通 generalist

  - 最省事。
  - 因为通才模型主要吃“单张图 + 指令”。
  - 你可以先参考 RoboDual/experiments/robot/openvla_utils.py 和 RoboDual/experiments/robot/robot_utils.py，直接让它在你的环境里出动作。
  - 这样最容易先验证任务描述、视觉输入、动作空间是否对得上。

  第二阶段：再接 dual system

  - 当你已经确认通才能工作，再接 RoboDual/vla-scripts/train_spacialist_calvin.py 里的 DualSystem 结构。
  - 这时你再决定是否要提供深度、夹爪视角、触觉这些额外模态。

  如果你接下来要训练自己的环境
  最快不是重写全套训练框架，而是“照着 CALVIN 数据格式喂数据”：

  - 优先复用 RoboDual/prismatic/vla/datasets/calvin_dataset.py
  - 让你自己的数据尽量长得像它现在读取的 npz + 语言标注
  - 至少准备：
      - rgb_static
      - rgb_gripper
      - depth_static
      - depth_gripper
      - robot_obs
      - actions / rel_actions
      - language
  - 这样你只需要改一个自定义 dataset loader，而不是改训练主干。

  我建议你的实际落地顺序

  1. 先实现一个 my_env_wrapper.py，接口模仿 RoboDual/vla-scripts/calvin_env_wrapper.py
  2. 先跑通通才模型推理
  3. 再接双系统评测控制器
  4. 最后再考虑数据采集和训练集格式

  如果你愿意，我下一条可以直接按“你的自定义仿真环境接入 RoboDual”这个目标，给你列一个最小实现清单，精确到“先新建哪几个文件、每个文件写什
  么类和方法”。


› 我是初次做类似的工作，你需要给我一些更加细致具体的指导：我原本理解的仿真是在solidworks类似的环境里面完成建模，然后有一个虚拟机械臂和
  虚拟摄像头。这是怎么和你上面说的“仿真环境包装”桥接起来的，我怎么0基础去操作


• 你原本的理解没有错，但还差一层“运行时接口”的概念。

  你可以把整件事分成两部分：

  1. SolidWorks / CAD 负责“把东西长什么样建出来”
  2. 仿真环境包装 负责“让程序能和这个世界交互”

  这两部分不是一回事。

  先说最关键的一句
  如果你是 0 基础、第一次做，最快路线不是从 SolidWorks 开始，而是：

  - 先用现成机械臂模型
  - 先用现成物体
  - 先在 Python 里的仿真器把场景跑起来
  - 最后再考虑把自己设计的 CAD 模型导进去

  因为 RoboDual 接的不是 “SolidWorks 文件”，而是“一个可交互的仿真程序接口”。

  ———

  你要建立的完整链条，其实是这样

  1. 你在 CAD 里画出物体或工装
  2. 把 CAD 模型导出成网格文件，比如 STL / OBJ
  3. 再把这些模型放进某个机器人仿真器里
     常见是 PyBullet、MuJoCo、Isaac Sim、Gazebo
  4. 仿真器里有：
      - 机械臂
      - 桌子
      - 物体
      - 摄像头
      - 物理碰撞
      - 任务初始化和重置逻辑
  5. 你再写一个 Python 包装层，把仿真器当前状态整理成 RoboDual 能读的格式
  6. RoboDual 输出动作，你再把动作翻译回仿真器的控制命令

  所以“桥接”发生在第 5、6 步。

  ———

  你可以把“仿真环境包装”理解成一个翻译器

  上游是你的仿真器，下游是 RoboDual。

  仿真器会给你这些原始信息：

  - 当前主相机 RGB 图
  - 当前夹爪相机 RGB 图
  - 深度图
  - 机械臂末端位置
  - 关节状态
  - 夹爪开合状态
  - 物体位置
  - 是否任务成功

  RoboDual 需要的是一个统一的 Python 字典，大致长这样：

  obs = {
      "rgb_obs": {
          "rgb_static": rgb_static_image,
          "rgb_gripper": rgb_gripper_image,
      },
      "depth_obs": {
          "depth_static": depth_static_image,
          "depth_gripper": depth_gripper_image,
      },
      "robot_obs": robot_state_vector,
  }

  以及一个环境类，至少有这些方法：

  - reset()
  - step(action)
  - get_obs()
  - get_info()

  这就是我前面说的“包装”。

  ———

  你现在最容易混淆的点
  很多初学者会以为：

  “我把机械臂和桌子在 SolidWorks 里搭好了，仿真就完成了。”

  其实不是。

  SolidWorks 更像是“建模工具”，不是 RoboDual 直接交互的“任务环境”。

  RoboDual 真正要对接的是“一个能运行的世界”，也就是：

  - 物体能掉下去
  - 机械臂能动
  - 相机能拍图
  - 你能 reset
  - 你能执行 action
  - 你能得到 observation

  这通常是在 PyBullet / MuJoCo / Isaac Sim 里完成的，而不是 SolidWorks 里。

  ———

  如果你完全 0 基础，我建议你按这个顺序做

  ### 第 1 步：别先碰 SolidWorks，先选仿真器

  对你这个项目，最现实的建议是：

  - 最快上手：PyBullet
  - 如果你后面想更强、更真实：Isaac Sim 或 MuJoCo

  但对于 RoboDual 这种 Python 代码结构，初学者最容易跑通的是 PyBullet。

  原因很简单：

  - Python 接口直接
  - 摄像头、深度图、物理步进都能拿到
  - 和 CALVIN 这类研究代码思路更接近
  - 调试成本低

  ———

  ### 第 2 步：先别做自己的 CAD 物体，先做“最小任务”

  比如只做一个最简单任务：

  - 桌子上有一个红色方块
  - 机械臂把它抓起来，放到一个目标区域

  这时你只需要：

  - 一个机械臂 URDF
  - 一个桌子
  - 一个立方体
  - 一个相机

  这样你能先理解：

  - 仿真器怎么跑
  - 观测怎么取
  - 动作怎么发

  ———

  ### 第 3 步：理解“动作”和“观测”是两个核心接口

  观测 observation
  RoboDual 主要关心的是：

  - 静态相机图像
  - 夹爪相机图像
  - 深度图
  - 机器人状态

  这些都不是“文件”，而是运行时的数组：

  - RGB 图通常是 H x W x 3 的 numpy.ndarray
  - 深度图通常是 H x W
  - 机器人状态通常是一串浮点数，比如：
      - 末端位置 xyz
      - 姿态 rpy
      - 夹爪开合

  动作 action
  RoboDual/ CALVIN 常用的是 7 维动作，近似可以理解为：

  - dx, dy, dz
  - droll, dpitch, dyaw
  - gripper

  也就是：

  - 前 6 维：让机械臂末端相对移动一点点
  - 最后 1 维：夹爪开/关

  所以你的仿真环境必须支持“给一个 7 维动作，让机械臂动一步”。

  ———

  ———

  那么 SolidWorks 在哪一步介入？
  是在“场景资产制作”这里介入，不是在“控制接口”这里介入。

  典型流程是：

  1. 在 SolidWorks 里建模一个零件，比如托盘、夹具、特殊物体
  2. 导出为 STL 或 OBJ
  3. 在仿真器里导入这个网格
  4. 给它补充物理属性：
      - 质量
      - 摩擦系数
      - 碰撞形状
      - 初始位置
  5. 放进任务场景里

  所以：

  - SolidWorks 解决“长什么样”
  - 仿真器 解决“怎么动、怎么碰、怎么拍”
  - 包装器 解决“怎么给 RoboDual 用”

  ———

  如果你要完全从 0 开始操作，我建议你别一上来做全流程，而是分 3 个里程碑

  ### 里程碑 1：单独把仿真器跑起来

  目标：

  - 机械臂出现
  - 桌子出现
  - 一个方块出现
  - 固定摄像头能拍图

  你现在不需要 RoboDual。
  你只要能在 Python 里打印：

  - RGB 图 shape
  - 深度图 shape
  - 机器人状态 shape

  比如确认：

  - rgb_static.shape == (256, 256, 3)
  - depth_static.shape == (256, 256)
  - robot_obs.shape == (7,) 或类似

  这一步是“证明你的世界可观测”。

  ———

  ### 里程碑 2：能执行动作

  目标：

  - 给一组动作，比如 [0.01, 0, 0, 0, 0, 0, 1]
  - 机械臂真的往前动一点，夹爪张开/闭合

  这一步是“证明你的世界可控制”。

  如果这一步做不到，后面一切都别谈。

  ———

  ### 里程碑 3：把它包装成 RoboDual 能调用的类

  目标：

  - 你的环境拥有 reset / step / get_obs
  - get_obs() 输出和 CALVIN 类似
  - 然后把这个环境塞给 RoboDual 的推理逻辑

  这一步才是“正式对接”。

  ———

  对你这种第一次做的人，我给你一个非常务实的建议
  不要先做“自己的新机械臂 + 自己的 CAD + 自己的摄像头系统 + 自己的任务协议”。

  这太重了。

  最快的入门方式是：

  - 先用一个现成机械臂模型，比如 Panda / UR5
  - 先用仿真器自带方块和桌子
  - 先实现一个简单 pick-and-place
  - 先只保留一个静态相机
  - 先让程序能返回假深度或者先不接专家模型
  - 先跑通通才推理

  也就是说，先证明“管线通了”，再慢慢提高真实性。

  你可以把整个项目拆成 4 层

  1. 资产层
      - SolidWorks 模型
      - STL/OBJ/URDF/USD/MJCF
  2. 仿真层
      - PyBullet / MuJoCo / Isaac Sim
      - 机械臂、物体、物理、摄像头
  3. 环境接口层
      - reset()
      - step(action)
      - get_obs()
  4. 策略层
      - RoboDual 通才模型
      - RoboDual 专家模型
      - 评测逻辑

  你现在最该学的是第 2、3 层，不是第 4 层，也不是一开始就沉迷第 1 层。

  ———

  如果你问“我第一周应该具体做什么”
  我会建议这样安排：

  1. 装一个最简单的 Python 仿真器环境
  2. 跑一个现成机械臂 demo
  3. 学会从虚拟相机拿 RGB 和深度图
  4. 学会读机器人末端位姿
  5. 学会发一个 7 维控制动作
  6. 写出自己的 MyEnvWrapper
  7. 用随机动作或手写动作测试 reset/step/get_obs
  8. 最后再接 RoboDual

  ———

  如果你问“我现在最不该做什么”
  最不该的是：

  - 一上来先做复杂 CAD
  - 一上来先想训练大模型
  - 一上来先重写 calvin_dataset.py
  - 一上来先追求真实感和工业级场景

  你现在最需要的是“可运行的最小闭环”。
