## codex 建议的实验顺序
先把日志按任务和结果分组：

  - success 轨迹 vs fail 轨迹。
  - failure 前最后 10/20 step vs 普通 step。
  - slow call 后第 0 到第 7 步分别统计。
  - 不同 subtask 分别统计，因为推块、开抽屉、按按钮的动作尺度和 gripper 行为会不同。

  看每个指标的 p50/p75/p90/p95/p99，不要只看均值。阈值初始可以从高分位开始，比如 sample_var > p90、dp_ref_l2 > p90、jerk_l2 > p95，再做
  组合规则。

  第三阶段：shadow trigger
  先实现“影子触发器”：每步根据这些指标判断“如果上线会不会触发 generalist”，但不真的改变调用逻辑。这样你能先估计：

  - 会触发多少次。
  - 会不会在正常成功轨迹上过度触发。
  - 失败前是否确实更容易触发。
  - 如果在固定 8 步刷新点选择跳过，会跳过多少次。

  注意：shadow trigger 不能完全预测动态策略效果，因为一旦你真的跳过 generalist，后续动作和观测分布会变；但它能筛掉明显没用的指标。

  第四阶段：小步在线实验
  再真正改变调度。不要一开始无限延长 generalist 间隔，建议做保守网格：

  - baseline：固定 8 步。
  - max_interval=10。
  - max_interval=12。
  - max_interval=16。

  规则可以是：默认不刷新，直到 max_interval 到达；如果任何风险指标超过阈值，就提前刷新。这样即使触发器漏判，也有最大间隔兜底。

  衡量指标不要只看成功率，还要看：

  - 平均成功序列长度。
  - 每个 episode 的 generalist call 数。
  - 控制频率 / model_s。
  - 不同 subtask 的成功率变化。
  - 失败发生在距离上次 slow call 第几步。

  一个代码层面的注意点
  不要继续依赖当前的 % 8 公式来做动态调度。现在 num_cond_actions 在 vla-scripts/dual_sys_evaluation.py:144 由全局 step 取模算出来，和固
  定 schedule 绑死。动态调度时应该显式维护 last_slow_step 和 steps_since_slow，再由它构造剩余 ref action。否则你会很难判断指标变化到底来
  自策略，还是来自 ref action 对齐错误。

  我会采用的实验顺序

  1. 跑固定 8 步 baseline，只记录日志。
  2. 离线画出 sample_var / dp_ref_l2 / jerk / gripper_flip / aggregation_delta 的分布和 failure 相关性。
  3. 做 shadow trigger，筛选 2 到 3 个最有效指标。
  4. 上线 max_interval=10/12 的保守触发器。
  5. 如果成功率基本不掉，再尝试 max_interval=16。
  6. 如果超过 8 步后掉得明显，再考虑只训练一个轻量 trigger/adaptor，而不是先重训 generalist。

## 开展了一次小数据集的测试
 选择了calvin_debug_dataset这是不是robodual的官方测试入口，官方测试入口的数据集大小高达571GB在个人电脑上完全没有办法完成

 ### 测试数据分析
  核心结果指标

  #### task_success / steps
  这是最重要的标签，来自 subtask_end。成功时 steps=step+1，失败时 steps=ep_len=240。你这批数据总体 313/381 = 82.15%，但分任务差异很大：
  close_drawer、move_slider_*、灯类任务基本稳定；stack_block、unstack_block、push_*_right、place_in_slider 明显差。
  价值：所有 profile 指标都应该最终和这个标签关联，而不是只看全局均值。

  #### step_success / terminal_step
  这两个在当前代码里等价：某个 step 被 oracle 判定完成当前 subtask。只有成功任务会出现 True，失败任务没有 terminal step，最后只写
  subtask_end(task_success=False)。
  价值：适合分析“成功前最后几步”的指标形态，但不能直接代表失败前状态。

  调度指标

  #### slow_system
  表示这一 step 是否调用 generalist/slow system。代码逻辑是 step == 0 或 (step + 1) % 8 == 0，也就是固定刷新，不是动态触发。你这里
  5501/42083 = 13.07% 为 True，符合约 8 步一次。
  价值：这是之后要降低慢模型调用次数的 baseline 参照。

  #### step_since_slow / step_since_slow_before
  表示距离上次 slow call 过去几步。step_since_slow_before 是本步 slow call 之前的值，step_since_slow 是本步可能调用 slow system 之后的
  值。
  价值：用它可以按“slow call 后第 0,1,...,7 步”分层看 dp_ref_l2、jerk、成功率。如果失败集中在 step_since_slow 较大时，说明 8 步
  reference 可能过期；如果没有明显上升，说明可以尝试延长间隔。

  #### num_cond_actions / ref_action_valid_part
  num_cond_actions 是当前从 generalist 8 步 action chunk 中还拿来给 specialist 条件化的有效动作数。ref_action_valid_part 就是这些有效
  reference actions。
  价值：这是解释 dp_ref_l2 的上下文。num_cond_actions 越小，reference 越接近 chunk 尾部；动态调度时这个逻辑不能继续死绑 % 8，否则
  reference 对齐会错。

  耗时指标

  #### model_s / profile.total_s
  二者基本重复，都是一次 model.step() 的总耗时。你这里 p50 约 0.156s，p90 约 3.52s，明显是 slow call 把尾部拉高。
  价值：衡量调度优化收益的主指标。

  #### fast_system_s
  specialist 一次预测耗时，包含 diffusion policy 的 forward 和可能的 sample_var 额外采样。fast step p50 约 0.14s，但有一部分到 0.41s 左
  右。
  价值：如果要优化控制频率，fast 系统自身也有波动，但不是主要瓶颈。

  #### slow_system_s
  generalist 调用耗时。你这里 p50 约 3.44s，p95 约 3.64s，第一步有 11.396s 冷启动/初始化异常。
  价值：这是降低 slow call 的直接收益来源。后续统计建议单独剔除每个 run 或每个进程的第一个 slow call。

  #### sample_var_s
  不是方差，是为了估计 sample variance 额外多采样的耗时。你的均值被少量 K=3 采样拉高，p50 是 0。
  价值：决定 sample_var 是否能在线用。它有诊断价值，但直接上线当每步触发器会增加成本。

  specialist 动作与 reference 差异

  #### dp_action_first
  specialist 当前预测的 8 步 chunk 里的第一个动作，7 维：x,y,z,roll,pitch,yaw,gripper。
  价值：适合人工看动作方向，但单独作为标量指标不够好。

  #### dp_action_chunk_mean / dp_action_chunk_l2_mean / dp_action_chunk_l2_max
  描述 specialist 预测的 8 步动作 chunk 的平均动作和动作幅度。它们不是误差，只是“动作有多大/多激进”。
  价值：如果失败任务里 l2_max 或某个维度频繁接近 ±1，可能说明动作饱和或过激；否则它更像辅助诊断。

  #### ref_action_first / ref_action_valid_part
  generalist 给出的 reference action。当前 specialist 的输入里同时有 ref_action 和 hidden_states。
  价值：主要用于和 dp_action 做差，不建议直接单看。

  #### dp_ref_l2 / dp_ref_l2_ee6 / dp_ref_l2_gripper
  这是非常有意义的指标：specialist 输出和 generalist reference 的 RMS 差异。ee6 是前 6 维末端动作，gripper 是夹爪单独通道。你这里
  dp_ref_l2_ee6 大多数很小，p95 约 0.0238，但尾部到 0.60；dp_ref_l2_gripper p50 很小但 p99 到 0.70，max 接近 2.0，说明整体异常常由
  gripper 二值翻转贡献。
  价值：dp_ref_l2_ee6 适合作为“specialist 是否偏离 generalist 太远”的触发指标；dp_ref_l2_gripper 要单独看，不能混到整体里，否则会掩盖
  ee6 运动质量。

  temporal aggregation 指标

  #### action_prediction
  最终真正送进环境的动作。它来自 temporal aggregation 后的动作，并且 gripper 被阈值化为 -1 或 1。

  #### raw_action_prediction
  aggregation 后、gripper 阈值化之前的动作。

  #### aggregation_delta / aggregation_delta_ee6
  最终动作 action_prediction 和当前新预测 dp_action[0,0] 的 RMS 差异。它衡量 temporal aggregation 把当前预测“改了多少”。
  你这里 aggregation_delta p95 约 0.30，p99 约 0.77；aggregation_delta_ee6 p95 约 0.175，p99 约 0.304。说明大多数 step 聚合变化不大，但
  有明显尾部。
  价值：如果失败前这个值升高，说明 specialist 当前动作和历史动作冲突，temporal aggregation 在“拉扯”输出，适合做风险指标。

  #### raw_aggregation_delta / raw_aggregation_delta_ee6
  和上面类似，但 gripper 阈值化之前计算。你看到 raw_aggregation_delta_ee6 和 aggregation_delta_ee6 完全一致，这是代码预期，因为阈值化只
  改 gripper，不改前 6 维。
  价值：用 raw 和非 raw 的差异可以判断 gripper 阈值化造成了多少额外跳变。

  平滑性与夹爪稳定性

  #### jerk_l2_ee6
  最终动作前 6 维的二阶差分范数，即动作是否突然改变。前两步没有历史，所以是 None。你这里 p50 约 0.066，p95 约 0.239，p99 约 0.420，max
  到 2.72。
  价值：很适合作为“动作不平滑/可能失控”的指标，尤其要按失败前窗口和任务类型看。

  #### gripper_jerk
  gripper 通道的二阶差分。因为最终 gripper 是 -1/1，所以值常常是 0, ±2, ±4。
  价值：不要和 ee6 jerk 混在一起；它更像“夹爪是否突然反复开合”的离散事件指标。

  #### gripper_flip_count
  最近 8 个最终 gripper 动作里符号翻转次数。你这里 p50 是 0，p90/p95 是 1，p99 是 2，max 是 5。
  价值：高 flip count 很可能对应夹爪犹豫或震荡，尤其对 lift_*、place_*、stack_block 很有诊断价值。

  不确定性指标

  #### sample_var / sample_var_ee6 / sample_var_gripper / sample_var_first
  这是对 specialist 重复采样 K=3 次后算的方差。但注意：代码里只有 step % 8 == 0 才算，而 slow call 是 step == 0 或 (step + 1) % 8 == 0。
  所以除了 step 0，后续 sample_var 基本是在 slow call 后下一步测的，不完全等于 slow call 当步的不确定性。
  价值：sample_var_ee6 是比较干净的不确定性指标；sample_var_gripper 和整体 sample_var 容易被夹爪主导。它很适合离线分析和 shadow
  trigger，但上线前要考虑额外耗时。

  观测变化指标

  #### obs_delta
  当前静态相机输入 tensor 和上一 step 的 RMS 差异。你这里分布相对稳定，p50 约 0.149，p99 约 0.310。
  价值：可以辅助识别场景变化、遮挡或大幅运动，但它不是直接动作质量指标。

  #### proprio_delta
  机器人状态前 6 维加 gripper 末维，相邻 step 的 RMS 差异。你这里 p50 很小 0.004，但 p95 到 0.756，p99 到 2.37，尾部很重。
  价值：可能对接触、夹爪切换、状态突变敏感；需要按任务和 gripper 状态拆开看，否则容易误解。

  我会优先保留的指标

  如果目标是“减少 slow model 调用，同时不掉成功率”，我会优先看：task_success/steps、slow_system/step_since_slow、dp_ref_l2_ee6、
  dp_ref_l2_gripper、aggregation_delta_ee6、jerk_l2_ee6、gripper_flip_count、sample_var_ee6、sample_var_gripper、model_s/slow_system_s/
  fast_system_s。

  相对不值得作为核心判断的有：rank、ep_len、runtime_dtype、sequence、subtask_i、oracle_s、to_numpy_s。它们主要是索引、常量或开销很小的工程字段。
