• evaluate_calvin_0428.py 相比 0424 版本，核心差异是：0424 只有固定 age 规则，0428 在它上面加了一层“变速 slow call 策略”。

  0424 的逻辑是：

  - fixed_mod8：第 0 步调用 slow system，之后按原始 (step + 1) % 8 == 0 固定刷新。
  - age_empty：第 0 步调用 slow system，之后只看 max_slow_age，到 age 上限才刷新。
  - empty_ref_after_age=8 后，reference action 置空。
  - 不会根据动作不稳定、jerk、sample variance、夹爪震荡等风险指标提前刷新。

  0428 的新逻辑是：

  - 新增高层参数 --slow_call_strategy。
  - 默认不是 0424 的 age_empty，而是 risk_balanced。
  - 仍然继承 0424 的空指导机制：slow_age_after >= empty_ref_after_age 时 reference 置空。
  - 但在 min_slow_age 之后，会根据“上一 step 的 profile 指标”判断是否提前 slow call。
  - 用上一 step 指标是因为当前 step 的 aggregation_delta_ee6、jerk_l2_ee6、sample_var_* 只有 fast 推理后才知道，不能在同一步提前知道。

  几个策略含义：

  - age_empty：等价于 0424 的 age_empty，不看风险，只到 max_slow_age 才刷新。
  - risk_balanced：主要推荐策略。age 到 8 以后，如果上一 step 出现明显风险，就提前刷新。
  - risk_conservative：更保守，只要任一风险指标超阈值就刷新，成功率更安全，但 slow call 会更多。
  - risk_score：多个风险指标累计成分数，达到 risk_score_threshold 才刷新。
  - risk_aggressive：更省 slow call，允许更晚刷新，风险更高。
  - fixed_mod8：回到原始 8 步固定 slow call。

  注意：在 0428 里，真正控制调度的是 --slow_call_strategy。--slow_trigger_policy 主要是底层兼容参数；只要 slow_call_strategy 不是
  fixed_mod8，代码会强制使用 age_empty 底层逻辑。

  我建议你先跑这几组：

  1. 0428 等价复现实验，先确认入口没破坏行为

  python vla-scripts/evaluate_calvin_0428.py \
    --dataset_subdir calvin_debug_dataset \
    --slow_call_strategy age_empty \
    --max_slow_age 12 \
    --empty_ref_after_age 8 \
    --profile_sample_var_ages 8,9,10,11 \
    --load_in_4bit \
    --low_cpu_mem_usage

  这组应该接近你之前 0425_1exp_command2。如果差很多，先不要测风险策略，说明 0428 入口或环境有变量。

  2. 主实验：risk_balanced

  python vla-scripts/evaluate_calvin_0428.py \
    --dataset_subdir calvin_debug_dataset \
    --slow_call_strategy risk_balanced \
    --max_slow_age 12 \
    --empty_ref_after_age 8 \
    --min_slow_age 7 \
    --risk_start_age 8 \
    --profile_sample_var_ages 8,9,10,11 \
    --load_in_4bit \
    --low_cpu_mem_usage

  这是最值得先看的策略：目标是保留 max_slow_age=12 的速度收益，同时在空指导阶段出现不稳定时提前刷新。

  3. 安全对照：risk_conservative

  python vla-scripts/evaluate_calvin_0428.py \
    --dataset_subdir calvin_debug_dataset \
    --slow_call_strategy risk_conservative \
    --max_slow_age 12 \
    --empty_ref_after_age 8 \
    --min_slow_age 7 \
    --risk_start_age 8 \
    --profile_sample_var_ages 8,9,10,11 \
    --load_in_4bit \
    --low_cpu_mem_usage

  如果这组成功率更高但 slow call 明显变多，说明风险指标确实能捕捉一部分失败前状态。

  4. 更省调用的试探：risk_score

  python vla-scripts/evaluate_calvin_0428.py \
    --dataset_subdir calvin_debug_dataset \
    --slow_call_strategy risk_score \
    --max_slow_age 12 \
    --empty_ref_after_age 8 \
    --risk_score_threshold 2 \
    --profile_sample_var_ages 8,9,10,11 \
    --load_in_4bit \
    --low_cpu_mem_usage

  这组要求至少两个风险指标同时异常才刷新，通常比 risk_balanced 更省 slow call，但可能漏掉单一强风险。

  我暂时不建议你先跑 max_slow_age=16 或 risk_aggressive。你现有结果已经说明 16 步空指导会明显掉成功率，下一步应该先在 max_slow_age=12 附
  近找平衡点。跑完后重点看三件事：整体成功率是否接近 0424/0425_1，slow call 比例是否明显低于 0413，弱任务的成功率有没有比纯 age_empty
  max_slow_age=12 更稳。
