  1. 空指导逻辑本身没有明显错

  代码里空指导是在 vla-scripts/dual_sys_evaluation_0424test.py:191：

  if slow_age_after is None or slow_age_after >= self.empty_ref_after_age:
      num_cond_actions = 0
  else:
      num_cond_actions = max(0, self.temporal_size - int(slow_age_after))

  也就是 empty_ref_after_age=8 时：

  - slow_age_after = 0：8 个 ref action
  - slow_age_after = 1：7 个 ref action
  - ...
  - slow_age_after = 7：1 个 ref action
  - slow_age_after >= 8：空指导

  我直接查了四组 JSONL：

  - slow_age_after < 8 却空指导：0 次
  - slow_age_after >= 8 却非空指导：0 次
  - slow_age_before < 8 就 slow call：0 次
  - slow_age_before < 8 就 risk slow call：0 次

  所以目前看，不是“本来应该返回空指导时实际没有空”这个问题。

  2. 选择性 slow call 不会在 8 步前触发

  0428 的 risk 判断在 vla-scripts/evaluate_calvin_0428.py:320：

  if slow_age_before < self.min_slow_age:

  你的参数里 min_slow_age=7、risk_start_age=8，但 risk 用的是上一 step 的 profile。所以实际第一批 risk slow call 会发生在：

  - 当前 slow_age_before=9
  - 上一步 source_age=8

  3. “当前步数小于 8 时会不会执行动作？”要分两种理解

  如果你说的是“会不会执行环境动作”：会。每一个 env step 都会执行 specialist 输出的动作。slow system 只是提供 ref_action 和 hidden_states 条件，不是直接控制机器人。

  代码顺序是：

  1. 判断是否 slow call：vla-scripts/dual_sys_evaluation_0424test.py:255
  2. 如果 slow call，更新 self.action、self.hidden_states、self.last_slow_step：vla-scripts/dual_sys_evaluation_0424test.py:257
  3. 根据新的 slow_age_after 构造 ref actions：vla-scripts/dual_sys_evaluation_0424test.py:272
  4. specialist 预测 dp_action：vla-scripts/dual_sys_evaluation_0424test.py:315
  5. temporal aggregation 后返回真正动作：vla-scripts/dual_sys_evaluation_0424test.py:381

  如果你说的是“会不会在小于 8 步时重新调用 slow”：不会，日志已经验证没有发生。

  4. 为什么选择性调用可能反而差

  关键在这里：slow call 后并不是直接替换最终动作，而是替换 specialist 的条件输入。可是 temporal aggregation buffer、hist_action、上一帧动作历史还都保留着。

  所以 risk call 发生时，系统状态是：

  - 前几步可能已经进入空指导；
  - aggregation buffer 里还混着旧预测；
  - 当前 step 突然换入新的 generalist hidden state 和 8 个新 ref actions；
  - specialist 的当前预测和历史 aggregation 可能发生拉扯。

  固定 schedule 的刷新相位是稳定的，specialist 比较容易适应；risk schedule 虽然调用机制一样，但刷新点集中在 aggregation_delta、jerk、sample_var 已经异常的状态附近，属于“在不稳定时插入
  新的条件分布”。这不是数学上等价的。

  性。
