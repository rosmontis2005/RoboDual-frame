• 我查了一些相近方向的工作。结合你现在的结果，我觉得最核心的启发是：

  不要把 slow system 当成“随时可以重新插入的 8-step action chunk”。更稳的做法通常是让 slow
  system 只在语义层、子任务层、目标层起作用，而不是在低层动作轨迹中途硬切换。

  你现在遇到的问题，本质上像是：

  fast specialist 已经沿着旧 hidden/ref/temporal buffer 形成一条轨迹
  ↓
  某个 risk 指标变高
  ↓
  突然重新接入 generalist hidden_states + 新 ref_action chunk
  ↓
  specialist 当前输出、历史 temporal aggregation、hist_action、旧轨迹之间发生拉扯

  这不是 slow call 本身错，而是切入点和条件形式不够平滑。

  1. 高层规划不要直接替换低层动作
  RoboDual 本身是 generalist 给 high-level task understanding 和 discretized action
  output，specialist 做多步 action rollout；论文也强调 generalist 和 specialist 的互补性，
  而不是让 generalist 每一步直接控制机器人。RoboDual 说 specialist 是 diffusion
  transformer，多步 rollout conditioned on generalist 的高层理解和离散动作输出，带来效率和
  控制频率提升。来源：RoboDual arXiv。

  HAMSTER 这类新 hierarchical VLA 更进一步：高层 VLM 不输出精细低层动作，而是输出粗 2D end-
  effector path，再由低层 3D-aware control policy 做精细控制。它明确说这样可以减轻高层 VLM
  预测 fine-grained action 的负担，也减轻低层 policy 的复杂推理负担。来源：HAMSTER arXiv。

  对应到你这里：
  与其在 age 9/10/11 硬切换一整段 ref_action + hidden_states，不如测试一种更软的形式：

  - generalist refresh 后不立刻完全替换 ref；
  - 只取新 generalist 的粗方向、末端 waypoint、gripper intent；
  - 或者对新旧 ref 做 2-3 步 ramp blending；
  - hidden state 切换也尽量放在固定边界，例如 age=12 或 subtask boundary。

  2. 其它快慢系统通常用“边界触发”，不是低层 jerk 触发
  SayCan 的思路是 LLM 负责高层语义，低层技能的 value function 负责 grounding。它不是根据低
  层动作 jerk 来重新规划，而是用“某个技能是否可行、是否推进目标”来选下一步技能。来源：
  SayCan paper summary。

  Inner Monologue 也是类似：它使用 success detection、scene description、人类反馈等环境反馈
  做 closed-loop replanning。重点是“任务进展/环境状态反馈”，不是短窗口动作不平滑指标。来
  源：Inner Monologue arXiv。

  对应到你这里：
  aggregation_delta、jerk、sample_var 更像“当前轨迹不稳定”的症状，不一定说明“现在应该换
  generalist”。更合理的 slow trigger 可能是：

  progress_stall over 16-24 steps
  or subtask oracle/proxy confidence low
  or object-state changed
  or gripper intent inconsistent for a sustained window
  or fast policy repeatedly outputs high-risk + no terminal progress

  也就是说，触发器应该从“动作形态异常”升级为“任务进展异常”。

  3. HiRT 的关键是慢特征低频更新，但低层实时连续执行
  HiRT 很接近你的问题。它把 VLM 放在低频，提取 temporally invariant features；高频 vision
  policy 用这些慢更新特征做实时控制。论文强调的是 trade-off：低频 VLM + 高频 policy，而不是
  在任意控制步强行重置低层轨迹。来源：HiRT arXiv。

  对应到你这里：
  你现在的 slow output 是 action chunk + hidden state，变化比较“硬”。可以考虑改成：

  - slow feature cache 持续使用；
  - ref_action 过期后置空，但 hidden_states 不随 risk 突然刷新；
  - 或者新 hidden_states 先 shadow 计算，不立即 commit；
  - 只有当新旧 specialist 输出一致性足够高时才切换。

  这类设计比“risk 一触发就换 slow guidance”更符合 HiRT 的低频慢特征思想。

  4. 动作 chunk 系统会特别重视 temporal consistency
  ACT / ALOHA 通过 action chunking 和 temporal ensembling 缓解 imitation learning 中误差累
  积、非平稳演示、动作抖动等问题。来源：ACT/ALOHA paper page。

  Diffusion Policy 也强调 receding horizon control、视觉条件和 time-series diffusion
  transformer。来源：Diffusion Policy arXiv。

  Diff-Control 更直接：它认为一致动作生成本身是一个难点，于是把 prior action trajectory 作
  为状态条件，让 diffusion policy 保持 temporal consistency。来源：Diff-Control project。

  对应到你这里：
  你的 temporal aggregation buffer 在 slow refresh 时没有重置，也没有显式对新旧条件做连续性
  约束。所以 risk refresh 后出现更高 aggregation_delta 是合理的。

  可以测试几个很具体的方案：

  1. slow refresh 后 ref ramp-in
      - 第 0 步新 ref 权重 0.25
      - 第 1 步 0.5
      - 第 2 步 0.75
      - 第 3 步 1.0
  2. slow refresh 后保留旧 ref 一两步
      - 新 hidden_states 先更新；
      - ref_action 仍用旧 chunk 或空 ref；
      - 过 1-2 步再使用新 ref。
  3. commit gate
      - slow model 可以提前算；
      - 但只有当 dp_action(new_ref) 和 dp_action(old_or_empty_ref) 的 ee6 差异小于阈值时才
        commit；
      - 否则继续空指导，等 max_age 再硬刷新。
  4. partial temporal buffer reset
      - 不完全清空 aggregation buffer；
      - 但降低旧 buffer 权重，例如 refresh 后旧预测权重乘 0.5；
      - 避免旧轨迹和新 ref 强烈冲突。

  5. RFST 给出的启发：选择系统应该基于任务类型，而不是每步震荡
  RFST 用 instruction discriminator 判断任务该走 fast 还是 slow thinking，慢系统用于需要意
  图识别和视觉推理的任务。来源：RFST arXiv。

  对应到你这里：
  你已经看到 hard tasks 和 easy tasks 对空指导敏感性不同。也许不该所有任务共用同一个
  max_slow_age=12：

  - drawer、slider、light 这类强任务：max_slow_age=12/13
  - stack、unstack、push_right、place_in_slider：max_slow_age=8/10
  - lift/table 类可能可用 max_slow_age=12，但 gripper flip 风险需要单独处理

  这比统一 risk trigger 更可能提升整体 trade-off。

  我建议优先尝试的三个方向

  1. 任务分组 age policy
     不改模型，只基于 task name 或 baseline difficulty 设置 max_slow_age。这是最现实的。
  2. risk 不直接刷新，而是 shadow refresh + delayed commit
     risk 触发后先调用 slow，但不立即替换当前指导；到 age=12 或动作一致性通过再切换。
  3. 新旧 ref 平滑切换
     重点解决你现在观察到的 refresh event 周围 aggregation_delta 和 action jump 变大的问
     题。

  我目前不建议继续单纯调 risk_start_age 或 risk 阈值。已有数据说明这些指标能找到“不稳定状
  态”，但还没有证明“在不稳定状态立刻切换 slow guidance”是正确动作。下一步应该把问题从“何时
  call slow”改成“slow 结果如何安全接入”。

  也就是说，不要推翻 task_age；它是目前最像“速度和准确率同时可控”的方向。

# 替换的思路方案
task_age 手动分组证明了“不同任务确实需要不同 slow/fast 协同模式”，但它本身不是一个有泛化能力的最终方案。 它更像一个
  oracle upper-bound 或诊断工具，而不是可部署策略。

  我建议把它定位成下一阶段的“监督信号来源”。

  核心思路
  不要把手动规则当成最终策略，而是用它回答两个问题：

  1. 哪些状态/任务应该更频繁接入 generalist？
  2. 什么时候接入 generalist 会破坏 specialist 的连续性？

  然后把这些经验转成一个可学习或可泛化的调度器。

  方向一：从 task rule 走向 task-conditioned scheduler
  第一步可以不是完全在线 trigger，而是做一个轻量 task-conditioned policy：

  输入：language instruction / task embedding / 初始图像特征 / 当前 subtask name
  输出：max_slow_age 或策略类别

  例如输出：

  easy/robust -> max_age=13
  default -> max_age=12
  sensitive -> max_age=10
  high-guidance -> max_age=8

  这比手写任务名规则泛化更好，因为它可以基于语言和视觉语义，而不是 hard-coded task string。你现在的 task_age 结果可以作为 pseudo-label。

  方向二：trigger 不应该直接“即时换指导”，而应该分成 detect 和 commit
  之前 risk trigger 失败的关键不是“检测不稳定”没用，而是：

  检测到不稳定 -> 立刻替换 slow guidance

  这个动作太硬。

  未来更合理的是两阶段：

  risk detector: 判断是否需要请求 slow system
  commit gate: 判断 slow result 是否可以安全接入

  也就是说，slow 可以提前 shadow call，但不要马上替换当前 hidden_states/ref_action。只有当：

  new-guided dp_action 和 current/empty-guided dp_action 差异不大
  或 temporal buffer 冲突小
  或当前轨迹已经自然到达 chunk 边界

  才 commit 新指导。

  这能弥合你现在遇到的问题：系统可以自主请求 slow，但不会因为“即时换指导”破坏动作连续性。

  方向三：把目标从“触发 slow call”改成“选择 guidance mode”
  现在我们只在做二选一：

  call slow / not call slow

  但真正的问题可能是多模式选择：

  full guidance: 使用 hidden_states + ref_action
  hidden-only: 更新 hidden_states，但 ref_action 置空
  ref-only or blended-ref: 使用新 ref，但逐步混入
  empty guidance: 完全靠 specialist
  fixed refresh: 到边界再刷新

  这比单纯 slow trigger 更合理。尤其你已经看到空指导并不总是坏，坏的是某些任务、某些阶段、某些接入方式。

  方向四：用离线数据训练/拟合调度器
  你现在已经有大量 JSONL，包括：

  - task success
  - steps
  - slow_age
  - task group
  - aggregation_delta
  - jerk
  - sample_var
  - gripper_flip
  - dp_ref_l2
  - empty_ref 状态
  - refresh 前后窗口

  这些可以构成一个离线数据集。可以先做很轻量的模型，不需要大训练：

  Logistic regression / decision tree / random forest / small MLP

  预测目标可以是：

  当前状态继续 empty/ref 走 k 步是否高风险
  当前 step refresh 后是否容易动作突变
  当前任务适合哪个 max_age 档位

  这样你不是凭直觉设阈值，而是从已有实验中学习调度边界。

  我认为下一阶段最合理的实验路线

  1. 继续用 task_age 作为 oracle-style baseline
     目标：证明“策略选择空间”确实存在收益。
  2. 做 task_age v2
     不是为了最终方案，而是把 oracle baseline 做得更强，给后续学习器一个更好的上界。
  3. 离线训练/拟合一个 task-conditioned scheduler
     输入先用 task language + 初始 obs + 简单任务统计，输出 max_age 档位。
  4. 再做 online adaptive scheduler
     但它不直接 commit slow result，而是：

     detect -> shadow slow call -> commit gate -> smooth/blend

  5. 最终比较四条线

     fixed_mod8
     uniform age_empty max12
     hand-written task_age
     learned scheduler / safe-commit scheduler

  如果从论文叙事角度看，task_age 的意义是：
  证明 RoboDual 这类 dual-system 的最优协同频率是 task-dependent 的；统一固定调度是次优的。

  后续真正贡献应该是：
  设计一个可泛化的 scheduler，让系统从语言/视觉/状态风险中预测何时以及如何接入 slow guidance，而不是手写任务表。

  所以你现在不是走偏了。手动规则不是终点，但它是很有价值的“oracle 证据”和“监督标签来源”。
